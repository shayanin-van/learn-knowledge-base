# Session 2026-05-14

## What was discussed

- Reviewed current architecture summary (hybrid curated wiki + RAG)
- Clarified "chunk" in practice for this system: since RAG runs on curated pages (not raw PDFs), chunks are sections of concept pages — clean semantic units, not mid-derivation cuts
- Discussed whole-page vs. section-level vs. hierarchical chunking:
  - Whole page = simpler, good starting point
  - Section-level = better retrieval precision, but sections-only context
  - Hierarchical (parent-doc retrieval) = section-level retrieval + full page context — the planned approach, already decided in architecture
  - Confirmed this is the same as Dense Hierarchical Retrieval / Document Summary Index already in the architecture decision
- Started Phase 3: concept page format design

## Decisions made

- **Concept page format**: Fixed metadata shell + free-form content body
  - Shell: summary, curriculum anchor (company structure), level, prerequisites, sources, last updated
  - Body: author-decided structure — no mandatory sections
  - Simulation block: parameterized URL reference (MVP); forward-compatible with stateful simulation linking
- **LaTeX** as math notation standard (platform standard)
- **IPST** is conflict-resolution authority only — not a structural or organizational template; conflicts noted inline in content body
- **Company curriculum structure** is the organizational anchor — provided per chapter by project owner
- **Prerequisites** as explicit wiki-links — enables concept dependency graph for the AI application layer
- **Worked examples on separate pages** — default, subject to revision
- **One concept, one depth, one page** — separate pages for different level treatments
